# Backend Changes / Client Workarounds

## Рейтинг с фильтрами

Сейчас экран рейтинга проектов и студентов поддерживает выбор метрик, периодов, порога быстрых PR и диапазона дат на клиенте. Это сделано как временный клиентский слой поверх текущего metric backend.

### Что уже сделано на клиенте

- Рейтинг проектов и студентов пересчитывается локально по данным `metric/project/:id`.
- Чипсы примененных фильтров, экран `Фильтры`, сохранение шаблона в рамках текущей сессии и пояснения метрик уже работают без backend API.
- `Личная статистика` на рейтинге открывает экран персональной статистики для текущего зарегистрированного пользователя по данным профиля.

### Что сейчас является workaround

- Backend пока не отдает отдельный endpoint для рейтинга проектов/студентов по произвольному набору метрик, параметров и диапазону дат.
- Для snapshot-метрик `Commits`, `Issues`, `Pull Requests` клиент временно маппит их в ближайшие оцениваемые производные метрики:
  - `Commits` -> `Total Commits`
  - `Issues` -> `Issue Completeness`
  - `Pull Requests` -> `Pull Request Hang Time`
- Для `Code Churn` клиент использует локальную эвристическую формулу оценки, потому что backend сейчас не предоставляет отдельный grading-contract для этой метрики в рейтинге.
- Шаблоны фильтров пока не сохраняются на backend и живут только в памяти клиента в рамках текущей сессии.
- Диапазон дат и per-metric параметры применяются только на клиенте во время пересчета списка, а не как backend query/filter state.

## Детальные экраны статистики

Сейчас экраны `Коммиты / Issue / Pull Request / Быстрые PR / Рефакторинг / Владение / ДДН`, которые открываются по `Подробнее`, строятся на клиенте из уже загруженных snapshot-метрик текущего репозитория.

### Что уже сделано на клиенте

- Клиент декодирует из snapshot-данных расширенный набор GitHub-полей, если они есть в payload:
  - `title`
  - `number`
  - `state`
  - `labels`
  - `comments`
  - `html_url`
  - `commits`
  - `additions`
  - `deletions`
  - `changed_files`
- Переключение участника на детальных экранах проектной статистики работает как локальный client-side filter поверх уже загруженных snapshot-данных.
- Для личной статистики детальные экраны используют тот же raw snapshot project context, но по умолчанию фиксируются на текущем пользователе.

### Что сейчас является workaround

- Backend не отдает отдельный endpoint/contract для detail screens, поэтому клиент сам агрегирует:
  - списки коммитов
  - списки issues / PR
  - lifetime buckets
  - ownership distribution
  - weekday distribution
  - top files / top fast PR / top slow PR
- Если конкретный snapshot не содержит расширенные GitHub-поля, экран показывает безопасный fallback:
  - обобщенный title
  - пустые labels/comments
  - только базовые даты/автора без deep metadata
- iOS-клиент временно разрешает небезопасные HTTP-запросы через `NSAppTransportSecurity`, потому что текущие `PRODUCTION_BASE_URL` и `METRIC_PRODUCTION_BASE_URL` работают по `http://`, а не по `https://`.

### Что нужно реализовать на backend

6. Явный detail contract для snapshot-экранов статистики:
   - commits details
   - issues details
   - pull requests details
   - rapid pull requests details
   - code churn details
   - code ownership details
   - dominant weekday details

7. Отдельный backend query/filter state для детальных экранов проекта по участнику, чтобы participant selector не работал только как client-side filter.

8. Перевести основной API и metric API на HTTPS, чтобы убрать глобальное ATS-исключение из iOS-приложения.

## Настройки и профиль

### Что уже сделано

- Клиент начал использовать полный payload профиля:
  - `user.email`
  - `user.phone`
  - `user.fullName`
- Клиент использует существующие backend endpoints:
  - `PUT /profile/personal`
  - `PUT /profile/account`

### Что сейчас является workaround

- Экран обратной связи временно шлет письмо напрямую из клиента по SMTP через локальный `MailConfig.kt`.
- Это dev-only костыль:
  - секрет почты живет на клиенте
  - его нельзя оставлять в релизной сборке
  - пароль приложения можно извлечь из установленного приложения
- Чекбокс `Получать оповещения об изменении статуса проектов...` пока сохраняется только локально в preferences приложения.

### Что нужно реализовать на backend

9. Добавить в профиль пользователя серверное поле для настройки подписки/оповещений по изменениям статуса проекта и вернуть его в `/user/profile`.
10. Добавить безопасный backend endpoint для отправки обратной связи из приложения, чтобы убрать прямую SMTP-отправку с клиента и исключить хранение почтовых секретов в мобильном приложении.

### Что нужно реализовать на backend

1. Endpoint рейтинга проектов с параметрами:
   - выбранные метрики
   - per-metric params
   - глобальный диапазон дат

2. Endpoint рейтинга студентов с тем же набором параметров.

3. Явный server-side scoring contract для snapshot-метрик:
   - `Commits`
   - `Issues`
   - `Pull Requests`

4. Явный server-side scoring contract для `Code Churn`, чтобы убрать клиентскую эвристику.

5. API для сохранения и загрузки шаблонов фильтров пользователя.
